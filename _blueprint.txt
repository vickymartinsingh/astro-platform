KEY BUSINESS RULES TO ENFORCE IN CODE:
• Wallet writes ONLY via Firebase Cloud Functions - never from the browser (Firestore rule: allow write: if false)
• Billing runs server-side in Cloud Functions every second - never in the browser
• Payment verified server-side with Razorpay HMAC signature before any wallet credit
• Billing stops IMMEDIATELY when user disconnects - Firebase onDisconnect handler
• Every wallet change (credit or debit) MUST create a transaction record
• Check wallet balance before every session start; block if insufficient
• Astrologer must select at least one service (Chat/Call/Video) to go online
• Session request times out after 60 seconds - status becomes missed
• Commission calculated after discount: astrologerEarning = (discountedPrice × duration) × (1 - commission/100)

5.12  Disconnect & Edge Case Handling - Astrologer Side
Edge Case
System Behaviour
Astrologer's internet drops during session
Firebase onDisconnect listener fires on both sides → session.status=ended → billing stops → client notified 'Session ended due to connection loss'
Astrologer turns offline while in session
Current session is allowed to finish; new session requests are blocked while in session
Two clients request same astrologer simultaneously
First request's 60s timer is active → astrologer accepts → status=busy → second client sees 'Astrologer is busy' immediately
Astrologer ignores request (60s expires)
status=missed; astrologer stays online but responseRate calculated drops; system can lower ranking if reject rate is high
Client closes browser during chat
Firebase onDisconnect ends session → billing stops → no overcharge; astrologer sees 'Client disconnected' message
Network unstable (frequent drops)
Grace time: 5-10 seconds before declaring disconnect. Agora has built-in reconnect logic.
Back button pressed during session
Session stays active - show confirmation dialog 'Are you sure you want to leave the session? This will end the call.'
Page refresh during session
Check Firestore for activeSession record - resume if still active; prevent duplicate billing timers

5.13  Back & Refresh Handling
📌  Every session screen must handle browser back button and page refresh. Check Firestore for activeSession flag on page load. If a session is already active, resume it rather than creating a duplicate.

SECTION 7: CORE ENGINE - BILLING, SESSION, WALLET, PAYMENT
7.1  Session Lifecycle (6 States)
State
Trigger & Meaning
requesting
Client clicks Chat/Call; session record created in Firestore; astrologer gets popup with 60s timer
accepted
Astrologer taps Accept within 60 seconds; Agora channel joined (for calls); billing starts
active
Both users connected; Cloud Function billing loop deducting wallet per second
ended
Session closed - by user, by astrologer, by auto-end (wallet=0), or by admin force-end
rejected
Astrologer tapped Reject; client shown 'Request rejected'; astrologer stays online
missed
60-second timer expired without astrologer action; client shown 'Astrologer unavailable'; no billing; astrologer stays online
State machine: requesting → accepted → active → ended
            requesting → rejected
            requesting → missed (after 60s)

7.2  Billing Logic (Core Business)
Rate calculation:
ratePerSecond = pricePerMinute ÷ 60
Example: ₹60/min ÷ 60 = ₹1.00 per second
Example: ₹30/min ÷ 60 = ₹0.50 per second
Cloud Function runs every second for each active session:
Step 1: Fetch current wallet from Firestore
Step 2: If wallet ≤ 0 → call endSession(sessionId) → STOP
Step 3: Else → newWallet = wallet − ratePerSecond
Step 4: Update wallet in Firestore (atomic transaction)
Step 5: Create debit transaction record
CRITICAL: Billing runs in Firebase Cloud Functions (server-side) only. NEVER in the browser. User cannot manipulate it.

7.3  Commission & Earnings Formula
After every session ends:
sessionCost = ratePerSecond × durationInSeconds
adminEarning = sessionCost × (commissionPercent ÷ 100)
astrologerEarning = sessionCost − adminEarning
Example: User spends ₹100 | Commission 30% | Admin earns ₹30 | Astrologer earns ₹70
Commission applies on DISCOUNTED price:
If priceChat=₹40, discount=50%: client pays ₹20. Commission applies on ₹20.

7.4  Wallet Safety Rules - NEVER Break These
⚠️  Wallet can ONLY be written by Firebase Cloud Functions using the Admin SDK. Firestore rules must enforce: allow write: if false for the wallet field.
✅  Every wallet change (credit OR debit) MUST create a corresponding transaction record.
✅  Before every session start: verify wallet ≥ (ratePerMinute × 1 minute). Block if insufficient.
✅  Payment success verified server-side using Razorpay HMAC signature BEFORE any wallet credit.
✅  Never trust amounts sent from the frontend - always recalculate on the server.
⚠️  Never update wallet without transaction log - if something breaks you must be able to audit and debug.

7.5  Common Billing Bugs - Must Avoid
Bug
Cause & Fix
Double deduction
Multiple billing timers running (e.g. page refresh). Fix: activeChat=true flag in Firestore prevents second timer from starting.
Incorrect balance after refresh
Frontend cached old value. Fix: Always read wallet directly from Firestore on page load - never trust localStorage.
Free usage exploit
User sends messages after wallet=0. Fix: Check wallet BEFORE every message send attempt in Cloud Function - block if zero.
Overcharge on disconnect
Billing continues after user drops. Fix: Firebase onDisconnect handler fires immediately and calls endSession.
Fake payment success
Frontend reports success without verification. Fix: All wallet credits ONLY after HMAC signature check in Cloud Function.

Action

7.7  Firebase onDisconnect - Disconnect Protection
When user or astrologer goes offline (browser closed, network lost, app minimized):
Firebase Realtime Database presence + onDisconnect handler:
1. Set isOnline=false in users collection
2. If activeSession exists: set session.status=ended
3. Cloud Function billing loop detects status=ended: stops deducting wallet
4. Both users notified of session end
Result: No overcharging. Astrologer who was still connected: session is ended for them too.

8.4  Cloud Function Code Samples
createUser - fires on every new signup
exports.createUser = functions.auth.user().onCreate(async (user) => {
  const userCode = generateUniqueCode(); // 9-digit unique
  const config = await db.collection('settings').doc('config').get();
  const signupBonus = config.data().signup_bonus || 0;
  await db.collection('users').doc(user.uid).set({
    name: user.displayName || '', email: user.email,
    wallet: signupBonus, role: 'client', userCode,
    hasUsedFreeChat: false, hasUsedFreeCall: false,
    isOnline: false, isOnCall: false, isBlocked: false,
    hasSeenTour: false, status: 'active', createdAt: new Date()
  });
  if (signupBonus > 0) { /* create credit transaction */ }
});

verifyPayment - Razorpay HMAC verification
exports.verifyPayment = functions.https.onCall(async (data, context) => {
  const { orderId, paymentId, signature, userId, amount } = data;
  const keySecret = functions.config().razorpay.secret;
  const expected = crypto.createHmac('sha256', keySecret)
    .update(orderId + '|' + paymentId).digest('hex');
  if (expected !== signature) throw new Error('Invalid payment signature');
  // Signature valid - safe to credit wallet
  await addMoney(userId, amount);
  return { success: true };
});

billingEngine - per-session wallet deduction
exports.billingEngine = functions.pubsub.schedule('every 1 minutes').onRun(async () => {
  const sessions = await db.collection('sessions')
    .where('status','==','active').get();
  for (const doc of sessions.docs) {
    const { userId, ratePerSecond } = doc.data();
    await db.runTransaction(async (t) => {
      const userRef = db.collection('users').doc(userId);
      const user = await t.get(userRef);
      const wallet = user.data().wallet;
      if (wallet <= 0) { await endSession(doc.id); return; }
      const deduct = ratePerSecond * 60; // 60 seconds per minute
      t.update(userRef, { wallet: wallet - deduct });
      t.set(db.collection('transactions').doc(), {
        userId, amount: -deduct, type: 'debit',
        reason: 'session', referenceId: doc.id,
        createdAt: new Date() });
    });
  }
});

8.5  Important Design Rules
⚠️  NEVER write Firebase code directly in the UI components. Always call a service function. Keep all business logic inside service files.
⚠️  NEVER directly update wallet from the frontend. Always go through walletService which calls Cloud Functions.
⚠️  NEVER mix chat logic and billing logic in the same component. Keep them separate services.
✅  Service functions are reused across client-web, astro-web, and admin-web - no duplication.
✅  All Firestore writes should use atomic transactions (db.runTransaction) when modifying wallet + transaction at the same time.

12.3  Firestore Security Rules (Production-Ready)
rules_version = '2';
service cloud.firestore {
  match /databases/{db}/documents {
    // Users: only read/write own record
    match /users/{uid} {
      allow read, write: if request.auth.uid == uid;
    }
    // Transactions: Cloud Function writes only - NEVER frontend
    match /transactions/{id} {
      allow read: if request.auth != null;
      allow write: if false; // Cloud Function uses admin SDK
    }
    // Astrologers: anyone can read approved astrologers
    match /astrologers/{id} {
      allow read: if true;
      allow write: if request.auth != null;
    }
    // Chat messages: only session participants
    match /chats/{chatId}/{sub=**} {
      allow read, write: if request.auth != null;
    }
    // Sessions: auth users only
    match /sessions/{id} {
      allow read, write: if request.auth != null;
    }
    // Settings: admin only writes; all can read for config
    match /settings/{doc} {
      allow read: if true;
      allow write: if false; // Admin SDK only
    }
  }
}
